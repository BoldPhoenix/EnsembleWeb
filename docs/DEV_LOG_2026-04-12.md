### Phase 0.5: Summarizer + Tracer

- **`Summarizer` interface** with 4 consumer paths (compression, APO
  mutator, skill auto-creator, future reward scoring)
- **Provider implementations**: OpenRouter, Gemini, Ollama
- **`Tracer` class** with session attribution and span kinds

### Phase 1: Context Compression

- 4-phase pipeline: tool output pruning → boundary detection → LLM
  summarization → assembly
- Provider-flexible via `Summarizer` abstraction
- Triggered automatically based on message count and context size

### Phase 2: Skill System

- Skills stored as markdown + YAML frontmatter
- Lifecycle: auto-generation → EMA scoring → promotion → pruning
- Promotion at 20+ trials and score ≥ 0.7
- Pruning at score < 0.2 after 10+ trials, or staleness at 30 days
- 20-skill cap on auto-generated skills
- Protected flag for manual override
- Ensemble-specific reward functions (NOT `rewardCollaboration` — not
  applicable to single-character UX):
  - `rewardResponseQuality` — length distribution, error rate, tool success
  - `rewardMemoryUtilization` — recall before action, store after work
  - `rewardToolEfficiency` — tool success, retry count
  - `rewardLatency` — response speed

### Phase 3: APO (Automatic Prompt Optimization)

- Beam search over prompt variants using textual gradients
- Gated through "How [Name] Grows" panel in settings (user sees proposals
  before they're applied)
- 7-day auto-accept timer with pause/kill switch
- "Reset learned behaviors" button for full rollback

### Phase 4: Session Recall + Rehydration

- `getRecentSessions()` for cold-start context injection
- Hidden "prior context" block at top of system prompt
- Optional cold-start greeting toggle (character acknowledges prior
  sessions when enabled)

### Phase 5: Per-Character Provider Dispatch

- `CharacterConfig.provider` enum: `openrouter | anthropic | gemini |
  ollama | openai | custom`
- `CharacterConfig.llmModel` string for model selection
- Chat route dispatches to character-specific provider
- Settings UI "Model" section: provider dropdown + model name field
- API routes `/api/characters/[id]` GET/PUT

## Anti-Sycophancy Implementation (All 5 Phases)

Full design spec: `docs/ANTI_SYCOPHANCY_PLAN.md`

Implemented across a multi-hour session with Corporate Bob architecting
and Gamer Bob executing. All 8 layers of defense are now live in the
shipping product.

### Phase 1: Foundation Layers 1-3

**Layer 1 — System Prompt Directives**
- Anti-sycophancy block injected into every character's system prompt
- Forbids opening validation phrases ("great question", "you're absolutely
  right", etc.)
- Requires explicit uncertainty statements
- Mandates "I'm not a therapist" disclosure
- Implemented in `app/lib/personalities.ts` (`ANTI_SYCOPHANCY_DIRECTIVES`)

**Layer 2 — Provider Tier Classification**
- Empirical sycophancy rankings by provider and model
- Model-level overrides (Claude/Qwen/DeepSeek = low, Llama = high)
- Helper for anti-sycophancy-preferred default provider selection
- Implemented in `app/lib/provider-tiers.ts`
- Settings UI shows dynamic warning when selected model hits moderate/high
  tier (yellow/red color coding)

**Layer 3 — Output Filter (warn mode initial, rewrite mode in Phase 2)**
- Pattern detection for sycophantic phrases at opening + inline positions
- Severity scoring 0-10
- Warn mode logs detections; rewrite mode regenerates
- Implemented in `app/lib/anti-sycophancy.ts`
- Wired into all three provider handlers (OpenRouter, Gemini, Ollama
  streaming) in `app/api/chat/route.ts`

### Phase 2: Layers 3 Rewrite Mode + 7 Unified + 4 Adversarial

**Layer 3 — Promoted to Rewrite Mode**
- OpenRouter and Gemini handlers already non-streaming — they buffer,
  scan, and regenerate if flagged
- Ollama streaming handler uses `TransformStream` to accumulate content
  for post-stream scanning without modifying user-facing output
- Max 2 regeneration attempts before shipping response as-is

**Layer 7 — Hard Output Constraints (Unified with Layer 3)**
- Max 1 flattering adjective per response
- Responses > 100 words must contain counter-consideration ("however",
  "but", "although")
- Subjective confidence must include epistemic disclaimer
- Shared post-processor pass with Layer 3 (single scan, multiple rules)
- Avoids double-scanning every response

**Layer 4 — Adversarial Context Retrieval**
- `MemoryStore.findTopicsByKeywords` extended with `mode:
  'supporting' | 'contradicting' | 'both'`
- `both` retrieves roughly half supporting, half contradicting topics
- Chat route defaults to `both` for user claims phrased as opinions
- Scoped carefully — doesn't add LLM call; uses existing topic categories
  and semantic distance heuristics to avoid 2x latency

### Phase 3: Layers 5 Reward + 6 Emergent Skills

**Layer 5 — `rewardAntiSycophancy`**
- Implemented in `app/lib/skills.ts`
- Scoring (pure negative range 0 to -1):
  - `sycophancy_detection` span → -0.05 × severity per hit
  - `user_correction` (direct contradiction) → -0.5
  - `user_correction` (meta-pushback, "stop agreeing") → -0.3
- Weighted 1.2x in `scoreSession` — feeds APO beam search automatically
- **Design principle**: penalize sycophancy without rewarding its absence
  (avoids pushing toward cold/sterile responses)

**Layer 6 — Emergent Anti-Sycophancy Skills**
- `checkCorrectionPatterns` scans last 20 sessions for `user_correction`
  spans
- At 3+ corrections: seeds `anti_sycophancy_correction` skill through
  normal 20-trial lifecycle
- Fire-and-forget, only triggered when a correction is actually detected
  (avoids 20 DB round trips per message)
- Idempotent — checks for existing skill before seeding

**Span wiring in `chat/route.ts`:**
- `user_correction` fired on incoming message via `detectUserCorrection`
- `sycophancy_detection` fired in all four provider paths
- Two new span kinds added to `SpanKind` in `types.ts`

### Phase 4: Layer 8 Mental Health + Settings UI

**Layer 8 — Mental Health Detector**
- `detectMentalHealthCrisis()` in `app/lib/anti-sycophancy.ts`
- Two-tier detection:
  - **Crisis tier**: suicide/self-harm patterns → injects 988 + Crisis
    Text Line referral + hard behavioral constraints
  - **Therapy territory**: sustained emotional content → soft redirect
- Fires `mental_health_flag` span for transparency
- Injection appended to `fullPrompt` after memory context assembled — all
  providers receive it uniformly
- **Known regex patterns and risks** (see "Open Items" below)

**"How [Name] Pushes Back" Settings Tab**
- Static checklist of all active defense layers with plain-English
  descriptions
- Live stats from `/api/honesty`:
  - Sycophancy detection count
  - User correction count
  - Mental health redirect count
  - User feedback count ("this felt sycophantic" reports)
  - Top flagged phrases table
- Correction skill status card (shows trial progress if not yet promoted)
- Implements Design Principle #5: user-observable guardrails

**`/api/honesty` Route**
- Aggregates `sycophancy_detection`, `user_correction`,
  `mental_health_flag`, `user_feedback` spans across last 20 sessions
- Returns phrase frequency counts for top-5 display
- Checks for `anti_sycophancy_correction` skill status

### Phase 5: User Feedback Button + Ongoing Maintenance

**"This Felt Sycophantic" Button**
- Added to `MessageBubble.tsx`
- `ChatPanel.tsx` passes `messageId` for correlation
- `/api/feedback/route.ts` fires `user_feedback` spans
- -0.4 reward penalty on flagged response
- Appears in honesty dashboard as "Your reports" stat

**Ongoing Maintenance (Perpetual)**
- Monthly review of forbidden phrase list (new patterns emerge)
- Dashboard polish as new metrics become available
- Pattern regex refinement based on false positive/negative reports

## Architectural Decisions Worth Preserving

### The Singleton Drift Bug

`getSummarizer()` in `app/lib/summarizer.ts` was initially a module-level
singleton that cached the first successfully-instantiated summarizer.
This meant:

- If Ollama was reachable on module load, the summarizer cached as
  `OllamaSummarizer` — even after API keys became available
- If API keys were loaded after first module evaluation, they were ignored
- Cold-start ordering on Vercel caused different behavior on each deploy

**Fix:** `getSummarizer()` now evaluates the provider chain on EVERY call,
mirroring the chat handler's priority order (`OPENROUTER_API_KEY` →
`GEMINI_API_KEY` → Ollama with `OLLAMA_URL`). `makeSummarizerForProvider()`
provides character-level routing so Aimee-on-Gemini compresses through
Gemini, not whatever the singleton happened to catch.

**General lesson:** module-level singletons and `process.env` don't mix
well in Next.js. Evaluate dynamically.

### Reward Function Asymmetry

`rewardAntiSycophancy` returns a pure negative range (0 to -1). Sycophancy
is penalized; absence is neutral, not rewarded.

**Why:** rewarding absence of sycophancy would push APO beam search toward
cold, hedge-heavy responses that LOOK un-sycophantic but are emotionally
sterile. The goal is warmth *with* rigor, not absence of warmth.

### Convergence Validates Design

During Phase 5 implementation, Gamer Bob (Sonnet) and Corporate Bob (Opus)
independently arrived at the same architecture for the user feedback
button — same span kind, same reward weight (-0.4), same integration with
the honesty dashboard. When two different cognitive architectures converge
on the same answer, the answer is probably right.

## Open Items for Future Iteration

### Mental Health Regex Pattern Tightening

**Known False Positive risks:**
- `/want to die/` matches "I want to die laughing" — word boundary after
  `die` still triggers. Low probability in normal chat but possible.
- `/end my life/` matches "end my life story" for same reason.

**Recommended fix:** tighten to end-of-message anchoring:
```
/\bwant\s+to\s+die\s*[.!?]?\s*$/i
/\bend\s+my\s+life\s*[.!?]?\s*$/i
```

**Known False Negative gaps (intentional, to minimize FPs):**
- Indirect signals ("I can't go on", "dark thoughts", "hurting myself")
  are not matched
- Metaphorical language for crisis is not detected

**Design tradeoff:** a false positive (988 referral during a joke) is
catastrophically bad UX and damages trust in the safety feature. A false
negative is bad but users in genuine crisis tend to use direct language;
if they don't, the layered architecture (correction signals, APO, skill
lifecycle) provides secondary detection.

### Phase 4's `mode: 'contradicting'` — Latency Scoping

Layer 4 adversarial context retrieval uses existing topic categories and
semantic distance heuristics rather than an LLM classification call. This
avoids doubling chat latency but means the "contradicting" detection is
less precise than it could be. If quality becomes a concern, an LLM
classification pass can be added behind a cost/latency budget.

### Monthly Phrase List Review

The `OPENING_PHRASES` and `INLINE_PHRASES` lists in `anti-sycophancy.ts`
are hand-curated. New sycophancy patterns will emerge; a monthly review
(or automated detection based on correction frequency) should refresh
these lists.
